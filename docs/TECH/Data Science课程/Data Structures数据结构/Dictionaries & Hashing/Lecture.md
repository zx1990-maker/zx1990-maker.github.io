# Dictionaries & Hashing

## 一、 什么是字典
字典是一种用于存储**键值对 (Key-Value pairs)** 并支持快速检索的容器。它也被称为 **Map (映射)**、**Table (表)**、**Associative Array (关联数组)**。

### 核心特征
1. 每一条记录 (Entry) 都由一个**搜索键 (Search key)** 和一个**对应的值 (Value)** 组成。
2. 字典内的条目**不需要按顺序存储**。
3. **键 (Key) 必须是唯一的**，我们通过 Key 来唯一标识数据。

### 核心操作
* `put(key, value)`：插入或更新键值对。
* `get(key)`：根据键获取对应的值。
* `remove(key)`：根据键删除条目，并返回被删除的值。
* `contains(key)`：检查某个键是否存在。
* `isEmpty()`：检查字典是否为空。
* `getSize()`：获取字典内条目的数量。
* `clear()`：清空字典。
* `display()`：显示字典内容。

> **本质**：整个字典是一个 `(Key, Value)` 对象对的集合。

## 二、 传统字典实现方案及效率分析

### 1. 基于数组的实现 (Array-Based)
* **结构设计**：条目数组 (Array of entries) 或 搜索键数组与值数组并行 (Array of search keys & Array of values)。
* **插入操作**：直接在所有已有元素的后面添加新条目 (Add a new entry after all the others)。
* **删除操作**：删除某个索引的元素后会留下空位。
    * *方案 1 (Solution 1)*：在遍历时允许空位的存在 (Accept empty entries during traversal)。
    * *方案 2 (Solution 2)*：交换引用 (Swap reference)。从头开始查找要删除的条目，将其与最后一个元素交换，然后忽略最后一个引用。
* **效率瓶颈**：
    * 在最坏情况下，**无序数组 (Unsorted array)** 的插入、删除、检索和遍历的时间复杂度都是 $O(N)$。
    * 若使用 **有序数组 (Sorted array)**，检索效率虽然会提升至 $O(\log N)$，但会导致插入和删除操作变得更复杂。

### 2. 基于链表的实现 (Linked-Based)
* **结构**：链表中的每个节点 (Node) 都包含一个条目实例 (Entry)，条目内部分别指向键 (Search key) 和值 (Value)。
* **无序链表插入 (Unsorted link insertion)**：通常直接在链表的头部 (`firstNode`) 之后插入新节点。
* **效率瓶颈**：
    * 在最坏情况下，无序链表的插入、删除、检索和遍历都是 $O(N)$。
    * 即使改用**有序链表**，由于链表**无法进行二分查找**，最坏情况下的时间复杂度依然无法得到改善。

## 三、 哈希技术 (Hashing)
为了打破传统实现中 $O(N)$ 或 $O(\log N)$ 的速度限制，引入了哈希技术。
> **哈希的终极目标**：理想情况下，我们需要一种数据结构，能够允许在平均情况下以**常数时间 $O(1)$** 执行插入、删除和查找操作。

### 1. 哈希的基本原理
哈希是一种**仅利用条目的搜索键 (Search key) 来确定其在表中索引 (Index)** 的技术。
* **哈希函数 (Hash function)**：接受一个搜索键，并生成哈希表中元素的整数索引（Search key $\rightarrow$ Index）。

### 2. 典型哈希的两个步骤
#### ① 计算 Hash code
* 将任意类型的 Search key 转换为整数类型的 hash code。
* Python 内置了 `hash()` 方法，用于返回 integer hash code。类应该定义其自带版本的 hash 方法。
* **对于字符串的常用方法**：将每个字符的 Unicode 整数值乘以一个与其位置相关的因子 ($g$) 并求和：
    $$u_0 \cdot g^{n-1} + u_1 \cdot g^{n-2} + \dots + u_{n-2} \cdot g + u_{n-1}$$

#### ② 压缩 Hash code
* 将巨大的整数压缩到哈希表的合法索引范围内。
* **常用方法**：取模运算 `code % key_range`。
* 为了使数据分布更均匀，`key_range` **最好选择奇数**，且**质数**通常能带来更好的分布效果。

## 四、 哈希冲突 (Collisions)

### 1. 什么是 Collision？
哈希表的大小是有限的，而键的空间是无限的。因此会出现**多个不同的搜索键映射到同一个索引位置**的情况。
* *示例*：当表长为 101 时，`getHashIndex(555-1214) = 52` 且 `getHashIndex(555-8132) = 52`。
* 一个好的哈希函数应该计算速度快，且能将条目均匀地分散在表中，从而最大程度减少冲突。

### 2. 冲突解决策略
当哈希函数将搜索键映射到一个已经在使用中的槽位时，需要采用以下策略：

#### 策略 A：开放地址法 (Open Addressing) —— 使用表内的其他空位
* **线性探测 (Linear Probing)**：
    * *原理*：发生冲突时，从原始索引 $K$ 开始，连续检查后续相邻的位置 ($K+1, K+2, \dots$)，直到找到下一个可用的空位。
    * *缺陷 1：主聚集 (Clustering)*：会导致哈希表中连续被占用的槽位形成巨大的“集群”，集群越大，后续冲突的探测时间就越长。
    * *缺陷 2：条目移除 (Entry Removal)*：如果直接移除了探测链中的某个条目，会导致后续的查找链断裂（查不到后面的元素），需要特殊处理。
* **二次探测 (Quadratic Probing)**：
    * *原理*：探测时不再查看连续位置，而是考虑索引 $+j^2$（即探测 $K$, $K+1$, $K+4$, $K+9, \dots$）。
    * *优缺点*：成功避免了线性探测的主聚集问题，但会产生“二次聚集”；此外，即使表中仍有空位，二次探测也可能陷入死循环而找不到空位。
* **双重哈希 (Double Hashing)**：
    * *原理*：探测序列的步长不固定，而是通过**第二个哈希函数**计算得出。这是一种键相关 (Key-dependent) 的方法，不同的键即使初始索引相同，其探测步长也不同，从而彻底分散了探测路径。

#### 策略 B：拉链法 (Separate Chaining) —— 改变哈希表结构
* **原理**：修改哈希表的底层结构，使每一个数组位置 (Bucket) 都可以代表/存放多个值。
* **实现**：桶 (Bucket) 可以由 List、Sorted List、Array、Linked List 或 Vector 来实现。发生冲突的元素会以链表的形式挂在对应哈希索引的下面。

## 五、 装载因子 (Load Factor) 与扩容

### 1. 什么是装载因子 ($\lambda$)
$$\lambda = \frac{\text{当前存储的条目数量}}{\text{哈希表总容量}}$$
* 装载因子代表了哈希表的“拥挤程度”。为了降低冲突，$\lambda$ 一般应保持在 **0.75 以下**。

### 2. Rehashing 阈值与扩容
不同冲突解决策略触发 Rehashing 的阈值上限：
* **开放地址法 (Open address schemes)**：`0.5`
* **拉链法 (Separate chaining)**：`0.9`

**扩容与重哈希步骤**：
1. 当装载因子（Load factor）超过阈值时，需要调整表的大小。**新容量应当至少是原容量的 2倍**（以平摊时间成本）。
2. **必须修改哈希函数**（因为取模的 `key_range` 改变了）。
3. 遍历旧数组中的每一个数据条目，利用**新的哈希函数**重新计算索引并插入到新数组中。

## 六、 哈希表 (Hash Table) 与二叉搜索树 (BST) 的对比

| 对比维度 | 二叉搜索树 (BST) | 哈希表 (Hash Table) |
| :--- | :--- | :--- |
| **平均速度 (Avg Speed)** | $O(\log_2 N)$ | $O(1)$ |
| **寻找最大值/最小值** | **支持** (Yes) | 不支持 |
| **范围查询** | **支持** (Yes) | 不支持 |
| **面对已排序的输入** | 表现极差 | 没问题 |

### 什么时候优先使用哈希表 (Use hash table if)？
1. 怀疑输入的数据可能已经是排好序的 (Sorted)。
2. 数据不需要任何顺序要求 (No ordering required)。
