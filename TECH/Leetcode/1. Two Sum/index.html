    1. Two Sum

    题目链接： https://leetcode.com/problems/two-sum/description/

    题目描述：

    Given an array of integers nums and an integer target, return indices of the two numbers such that they add up to target.

    You may assume that each input would have exactly one solution, and you may not use the same element twice.

    You can return the answer in any order.

    solution：

    class Solution(object):
    def twoSum(self, nums, target):
        """
        :type nums: List[int]
        :type target: int
        :rtype: List[int]
        """
        memo = {} # Key: 遍历过的数字, Value: 该数字的索引

        # 使用 enumerate 可以同时拿到索引 i 和具体的数字 num
        for i, num in enumerate(nums):
            complement = target - num # 计算当前数字需要的配对值

            # 检查配对值是否在我们之前存过的记录里
            if complement in memo:
                # 如果在，说明找到了！返回配对值的索引，和当前数字的索引
                return [memo[complement], i]

            # 如果不在，把当前数字和它的索引存入字典，继续往后找
            memo[num] = i

        return None

    algo:

    面对这类“需要在数据集合中进行高频查找或配对”的题目，我们在脑海中应该有一套标准的设计图纸。你可以按照以下四个核心步骤来思考和设计程序：

    1. 洞察暴力解法的“痛点” (Find the Bottleneck)

    拿到题目后，先不要急着想最优解，先想最笨的办法（暴力解法）是怎么做的。

    • 思考： 最笨的方法通常是两层甚至三层 for 循环。比如 Two Sum，最笨的方法是用指针 \(i\) 遍历数组，然后再用指针 \(j\) 遍历一遍剩下的数组去找配对。
    • 痛点： 这里的痛点就是“大量的重复查找”。每次都在做无用功。
    • 破局： 一旦你发现程序在不停地“回头找东西”，就应该立刻条件反射般地想到：我要用哈希表（字典）把走过的路记下来，实现 \(O(1)\) 的秒查。

    2. 确立“通缉令”：设计 Key

    字典的 Key 是哈希表设计的灵魂。你要问自己：“接下来我（或者后面的元素）究竟要找什么？”

    • 原则： 把你要找的那个“目标特征”作为 Key。
    • 举例： * 在 Two Sum 中，我们真正在找的不是当前的数字，而是能和当前数字凑成 target 的那个配对数字。所以 Key 设计为具体的数字（方便后面的配对值来查它）。
    • 在“字母异位词分组”中，我们要找的是“构成字母相同的单词”。所以 Key 设计为排序后的字符串。

    3. 确立“悬赏金”：设计 Value

    找到了 Key 之后，你需要用到什么信息？这就是 Value 应该存的东西。

    • 原则： 题目最终要求返回什么，或者下一步计算需要用到什么，Value 就存什么。
    • 举例：
    • Two Sum 题目要求返回索引，所以 Value 存索引。
    • 如果题目要求返回“有多少对数字之和等于 target”，那么 Value 就应该存这个数字出现的频次 (count)。
    • 如果要求对字符串进行分组，Value 就应该存一个 List，把属于这个 Key 的原单词都塞进去。

    4. 决定遍历与存储的“时序” (Timing)

    这一步决定了程序的逻辑结构，主要分为两派：

    • 流派 A：边遍历，边查找，边存入 (One-pass)
    • 适用场景： 需要找“配对”、且需要防止“自己匹配自己”。
    • 逻辑： 永远先去字典里查，查不到再把当前元素当成历史记录存入字典。
    • 流派 B：先全量统计，再统一处理 (Two-pass)
    • 适用场景： 找高频元素、查重、或者后续操作依赖于全局完整的统计数据。
    • 逻辑： 先用一个独立的 for 循环把所有东西一股脑塞进字典，然后再用第二个循环或方法去字典里捞数据。

    💡 总结成一句话心法：

    只要发现代码在做“嵌套遍历查找”，立刻停手，设计一个字典：用 Key 定义你要找的特征，用 Value 存储你要用的结果，通过空间换时间把嵌套循环拆成单层循环。
